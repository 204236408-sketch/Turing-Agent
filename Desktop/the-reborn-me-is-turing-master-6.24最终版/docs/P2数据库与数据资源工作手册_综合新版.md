> 角色定位：数据库设计、MySQL schema、种子数据、知识库、题库、视频资源、数据迁移、备份。  
> 上游：PDF 技术方案、差距审计、全量规则文档。

## 0. P2 无痛执行区

这一节是 P2 每天开始前先看的执行卡。P2 是最上游，P2 的交付如果不清楚，P3/P1/P4/P5 都会被迫猜字段、猜数据含义。

### 0.1 今天开始前先确认

| 要确认的事 | 找谁确认 | 目的 |
|---|---|---|
| 今天哪个业务链优先 | 组长/P5 | 避免补了很多数据但不是当天要验收的流程 |
| 后端模型字段 | P1 | 保证 `schema.sql`、JSON、`models.py` 不打架 |
| Agent 需要哪些上下文 | P3 | 知道知识点、错因、关键词、文档要写到什么粒度 |
| 页面需要哪些展示字段 | P4 | 例如视频封面、时长、理由、标签是否需要 |
| 验收数量和样例 | P5 | 明确今天至少多少条数据、哪些样例会被测试 |

### 0.2 P2 每天固定执行顺序

1. 上午先改表结构或 JSON 字段，不先堆数据。
2. 每改一个字段，立刻同步 P1 和 P3。
3. 中午前给 P3/P1 至少 1 条完整样例数据。
4. 下午补足数量，并同步修改导入脚本。
5. 16:00 前运行导入脚本，确认数据真的进入数据库。
6. 晚上把数据量、字段变化、导入命令写进交付说明。
7. 如果数据没达标，必须说明是数量不足、字段不稳，还是导入失败。

### 0.4 P2 完成标准

| 层级 | 完成标准 |
|---|---|
| 表结构 | `schema.sql`、迁移脚本、`models.py` 字段含义一致 |
| JSON | 字段名稳定，UTF-8 可读，至少有一条完整样例 |
| 导入 | 导入脚本能跑，重复执行不产生明显脏数据 |
| 数量 | 达到当天数量要求，并能被接口查询到 |
| 下游 | P3/P1/P5 都知道字段含义和导入方式 |

不算完成：

- 只写了 JSON，但导入脚本不支持。
- 只补了表，但没有说明字段给谁用。
- 数据量够了，但知识点名称混乱，导致推荐/RAG 对不上。
- 改字段后没有通知 P1/P3/P4/P5。

### 0.5 新 P2 接手时怎么做

1. 打开 `docs/验收清单.md`，先看数据相关失败项。
2. 打开 `backend/models.py` 和 `database/schema.sql`，确认表字段是否一致。
3. 跑一次导入脚本，确认当前数据能进库。
4. 抽查 3 条知识点、3 道题、3 条视频，确认字段完整。
5. 先修会阻塞 P3/P1 的字段和导入问题，再补数量。

### 0.6 P2 遇到阻塞怎么处理

P2 最常见阻塞是字段不确定、数据量来不及、导入脚本失败。不要把不稳定字段交给下游。

```text
阻塞点：
影响哪张表/哪个数据文件：
影响 P3/P1/P4/P5 哪个动作：
需要谁确认：
临时样例或 fallback：
最晚需要时间：
```

如果字段 30 分钟内定不下来，先交一条明确标注为“临时样例”的数据给 P3/P1 调通流程，晚上前再统一字段。

### 0.7 P2 完成后上传到 Gitee

按 `docs/Gitee提交流程.md` 上传。P2 通常只提交：

## 3. 数据命名和格式总规则

### 3.1 科目名称固定

只能使用：

```text
数据结构
计算机组成原理
操作系统
计算机网络
```

不能混写：

```text
计组
组成原理
OS
网络
```

### 3.2 知识点名称全局唯一

例如统一使用：

```text
页面置换算法
```

不要混用：

```text
页置换
LRU算法
页面替换
```

### 3.3 JSON 字段规则

所有 JSON 使用 UTF-8，字段名统一英文 snake_case。

布尔值：

```json
"is_high_frequency": true
```

数组字段：

```json
"keywords": ["LRU", "FIFO", "缺页次数"]
```

如果当前导入脚本暂不支持数组，P2 需要同步修改 `seed_data.py`，不能要求 P3/P1 猜格式。

```text

```

模板：

```md
# 页面置换算法

subject: 操作系统
knowledge_point: 页面置换算法
keywords: LRU,FIFO,OPT,Clock,缺页次数

## 核心概念
页面置换算法用于在发生缺页且物理页框已满时选择淘汰页面。

## 408 常见考法
- 给定访问序列计算缺页次数。
- 比较 FIFO、LRU、OPT 的替换过程。

## 解题步骤
1. 写出页框数量。
2. 按访问序列逐步模拟。
3. 命中时更新算法状态。
4. 未命中时计入缺页。

## 常见错误
- 初始装入没有计入缺页。
- LRU 命中后没有更新最近访问顺序。

## 例题
...
```

最低数量：

- 至少 20 份。
- 高频知识点优先。

### 下午要做：推荐和掌握规则

修改：

```text
backend/services/recommendation_service.py
backend/services/mastery_service.py
```

推荐评分：

```text
recommend_score =
  status_weight * 0.40
  + recency_weight * 0.20
  + frequency_weight * 0.15
  + weak_score_weight * 0.15
  + interval_weight * 0.10
```

掌握度 `weak_score`：

```text
wrong_count * 3
+ unfamiliar_count * 2
+ unknown_count * 4
+ ocr_mistake_count * 3
+ concept_error_count * 2
+ rule_confusion_count * 2
+ qa_count * 1
+ forum_count * 1
- correct_count * 1
- mastered_count * 2
```

知识点级状态优先级：

```text
无行为 -> 未学
weak_score >= 10 或 repeated_error_type >= 2 -> 薄弱点
unknown_count >= 1 或 correct_rate < 50% 或 continuous_wrong_count >= 2 -> 不会
unfamiliar_count >= 1 或 50% <= correct_rate < 80% 或 wrong_count in [1,2] -> 不熟
total_answer_count >= 3 且 correct_rate >= 80% 且 weak_score <= 2 -> 掌握
其他 -> 不熟
```

### 交付标准

- P3 可以导入知识库到 ChromaDB。
- 首页推荐排序有明确公式。
- 题本状态和知识点状态不再混用。
