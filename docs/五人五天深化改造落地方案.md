### 3.4 知识库向量化文本格式

文件目录：`backend/data/knowledge_docs/`

每个知识点一个 Markdown 文件，例如：`操作系统_页面置换算法.md`

```md
# 页面置换算法

科目：操作系统
父级：内存管理
关键词：LRU、FIFO、OPT、Clock、缺页中断

## 核心概念
页面置换算法用于在页框不足时选择被换出的页面。

## 408 常考方式
1. 给定访问序列计算缺页次数。
2. 比较 FIFO、LRU、OPT 的替换结果。
3. 判断 Belady 异常。

## 常见错因
1. 初始装入缺页漏算。
2. LRU 命中后未更新最近使用顺序。
3. FIFO 与 LRU 队列更新规则混淆。

## 标准回答模板
先列页框变化，再统计缺页次数，最后说明替换依据。
```

导入流程：

1. P2 准备 Markdown 知识库文件。
2. P3 修改 `backend/scripts/import_knowledge_to_chroma.py`，把 Markdown 拆分成 chunk。
3. 每个 chunk 元数据包含：`subject`、`knowledge_point`、`section`、`source_file`。
4. ChromaDB 不可用时，把全文写入 MySQL 表 `knowledge_document`，使用 `LIKE` 降级检索。

### 5.3 视频评分公式

```text
quality_score =
  platform_weight * 0.30 +
  relevance_weight * 0.40 +
  popularity_weight * 0.20 +
  freshness_weight * 0.10
```

评分规则：

- `platform_weight`：中国大学 MOOC 90，Bilibili 高质量 UP 85，Bilibili 普通视频 75，YouTube 70。
- `relevance_weight`：标题包含知识点 100，标题包含同义词 80，简介包含知识点 60，否则 30。
- `popularity_weight`：能拿播放量则按播放量归一化，拿不到默认 50。
- `freshness_weight`：一年内新视频最高，老视频但经典可人工给 70。

## 8. 流水线协作总则

新补充的后端流水线文档强调：每个功能不是某个人单独完成，而是一条从数据到前端再到测试的流水线。五个人必须按“上游产物交给下游”的方式协作。

### 8.1 总流水顺序

```text
P2 数据资源 → P3 Agent/算法 → P1 路由接口 → P4 前端交互 → P5 测试验收
```

每个角色只要记住一句话：

- P2 产出“干净可查的数据”。
- P3 把数据加工成“智能结果”。
- P1 把智能结果包装成“稳定接口”。
- P4 把接口变成“用户看得见、点得动的页面”。
- P5 验证整条链路“能跑、能错、能恢复”。

### 8.2 五条核心流水线

| 流水线 | 最终产品 | 上游原料 | 主要下游 |
|---|---|---|---|
| 答题闭环链 | 出题 → 答题 → 批改 → 错因 → 记忆 | 题库、知识点、掌握状态、LLM | 首页推荐、错题本、报告 |
| 问答链 | AI 问答 + Agent 步骤 + 记忆引用 | LLM、RAG、用户画像、会话历史 | 长期记忆、报告问答焦点 |
| 视频资源链 | 每道题关联 2-3 个推荐视频 | 种子视频、爬虫、知识点 | 出题页视频抽屉 |
| 论坛社交链 | 帖子、评论、AI 回答、排行榜 | 用户行为、点赞、LLM、画像 | 论坛页、排行榜、学习画像 |
| 报告生成链 | 学习报告 + 可点击训练计划 + 导出 | 所有学习行为数据 | 报告页、PDF/打印导出 |

### 8.3 答题闭环链交接协议

这条链是项目核心，必须最先打通。

```text
P4 用户点击生成题目
  → P1 `POST /api/questions/generate`
  → P3 `question_agent.generate_questions`
  → P2 写入 `question_generation_session` 和 `question`
  → P4 渲染题目

P4 用户提交答案
  → P1 `POST /api/answers/check`
  → P3 `answer_check_agent.check_answer`
  → P2 写入 `answer_record`，重算 `knowledge_mastery`
  → P4 展示批改结果和推荐错因

P4 用户确认错因
  → P1 `POST /api/mistakes/cause-confirm`
  → P3 `mistake_agent.confirm_cause`
  → P2 写入 `mistake`、`user_memory`，同步 Chroma 或降级检索
  → P4 展示“已写入长期记忆”
```

接口输入输出必须固定：

| 交接点 | 请求 | 必填字段 | 返回必须包含 |
|---|---|---|---|
| 生成题目 | `POST /api/questions/generate` | `mode, subject, knowledge_point, difficulty, question_type, count` | `session_id, questions[]` |
| 提交答案 | `POST /api/answers/check` | `question_id, user_answer` | `is_correct, feedback, answer_record_id, recommended_causes` |
| 确认错因 | `POST /api/mistakes/cause-confirm` | `answer_record_id, error_types[], user_note, evidence_source` | `mistake_id, memory_id, suggestion, weak_score_change` |
| 标记掌握 | `POST /api/questions/{id}/mastery` | `status` | `status, weak_score, triggered_actions[]` |

### 8.4 问答链交接协议

```text
P4 输入问题
  → P1 `POST /api/qa/chat`
  → P3 `qa_agent.answer_question`
  → P2 写入 `conversation`、`conversation_message`，更新 `qa_count`
  → P4 展示结构化回答和 Agent 步骤
```

`qa_agent` 必须返回：

- `subject`
- `knowledge_point`
- `answer`
- `agent_steps`
- `memories_used`
- `suggested_followups`
- `llm_used`

追问规则：

- 用户输入短句，例如“为什么”“举个例子”“那 TCP 呢”，P3 必须继承上一轮 `subject` 和 `knowledge_point`。
- P1 不新建会话，只复用当前 `conversation_id`。
- P4 在历史会话中展示摘要，而不是只截取前 30 个字。

### 8.5 视频资源链交接协议

```text
P2 先准备 `seed_video_resources.json`
  → P3 爬虫补充 `video_resource`
  → P1 `GET /api/videos/recommend`
  → P4 视频抽屉展示 top 3
  → P5 验证无视频、少视频、爬虫失败三种情况
```

关键要求：

- P2 必须保证每个高频知识点至少 2 条人工审核视频。
- P3 爬虫只能补充，不能让用户请求长时间等待。
- P1 返回字段必须包含 `items`、`crawl_triggered`、`fallback_used`。
- P4 不再用 toast 一条条弹视频标题，必须展示完整视频卡片。

### 8.6 论坛社交链交接协议

```text
P4 发帖/评论/点赞/打卡
  → P1 `forum_router`
  → P2 写入 forum 相关表和 user_daily_activity
  → P3 论坛 AI 读取帖子作者画像生成个性化回答
  → P4 展示 AI 回答、排行榜、热帖
```

P1 必须保证：

- 点赞和收藏按用户幂等，不允许重复加数。
- `GET /api/forum/posts` 支持 `q/search`、`category`、`page`、`size`。
- `GET /api/forum/leaderboard` 可按 daily/weekly 返回榜单。

P3 必须保证：

- 论坛 AI 回答要读取帖子作者的 `knowledge_mastery`、`user_memory`、`mistake`。
- 无 LLM 时返回通用回答，但结构仍然一致。

### 8.7 报告生成链交接协议

```text
P4 点击生成报告
  → P1 `POST /api/reports/generate`
  → P3 `report_agent.generate_report`
  → P2 聚合 answer_record、knowledge_mastery、mistake、user_memory
  → P4 展示报告页
  → P1 导出 PDF 或降级打印
```

报告计划必须是结构化数据，不允许只返回纯文本：

```json
{
  "step": 1,
  "subject": "操作系统",
  "knowledge_point": "页面置换算法",
  "action": "专项练习",
  "count": 3,
  "difficulty": "中等",
  "estimated_minutes": 25,
  "reason": "针对 LRU 缺页统计遗漏"
}
```

## 11. 验收主流程

必须逐条通过：

1. 注册/登录演示用户。
2. 首页显示今日计划、倒计时、推荐、统计、记忆。
3. 点击“开始个性化训练”自动生成题目。
4. 答题后返回结构化批改。
5. 答错后展示推荐错因。
6. 确认错因后写入错题和长期记忆。
7. 错题本按“不熟/不会”展示对应题。
8. 错题详情能看到相似错题。
9. 视频抽屉展示 2 条以上推荐视频。
10. QA 问答返回结构化回答和 Agent 步骤。
11. 论坛发帖、评论、点赞、取消点赞、排行榜可用。
12. 论坛 AI 回答能结合提问者学习画像。
13. 报告页展示真实统计、趋势、画像、权重、训练计划。
14. 报告可导出。
15. MySQL 数据库重启后数据仍在。
16. 云服务器访问正常。
