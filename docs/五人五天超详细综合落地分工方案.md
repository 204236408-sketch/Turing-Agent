## 2. 总体目标与不可妥协标准

### 2.1 五天结束的最终效果

五天结束后必须达到：

1. 本地可运行，云端可访问。
2. 云端业务数据库使用 MySQL，SQLite 只作为本地兜底。
3. ChromaDB 至少有三个 collection：`knowledge_base_408`、`user_memory_vector`、`mistake_summary`。
4. QA/RAG、智能出题、答题批改、错题分析至少主链路使用 LangGraph 编排，保留原函数名给路由调用。
5. 数据资源达到最低规模：
   - 4 科。
   - 40+ 知识点。
   - 60+ 题目。
   - 80+ 视频资源。
   - 20+ 知识库 Markdown。
6. 首页、问答、出题、错题本、OCR、论坛、报告主流程使用真实接口。
7. 视频推荐按 `subject + knowledge_point` 返回 2-3 条资源，爬虫失败时有种子视频兜底。
8. “不熟/不会题本”按题目级状态管理，“薄弱点”按知识点级标签管理。
9. 论坛点赞收藏幂等，排行榜和热度排序可用。
10. 学习报告使用真实统计，训练计划可点击进入出题。
11. 所有外部服务失败时有 fallback：无 LLM、无 ChromaDB、视频爬虫失败、OCR 不可用。
12. P5 输出最终 `docs/验收清单.md` 和 `docs/上线风险清单.md`。

### 2.2 最小可上线边界

如果五天内有风险，必须优先保证：

1. MySQL 可部署、可导入、可备份。
2. 首页 + 出题 + 答题 + 错因 + 错题本完整闭环。
3. QA/RAG 可运行，Chroma 不可用时有关键词降级。
4. 核心 Mock 去除，不用固定假数据冒充真实结果。
5. 视频先用 80 条种子资源兜底，真实爬虫可作为增强。
6. 测试和验收清单完整。

## 3. 五人流水线协作模型

### 3.1 总顺序


### 3.2 五条核心流水线

| 流水线 | 上游输入 | 中间处理 | 下游输出 | 负责人链 |
|---|---|---|---|---|
| L1 答题闭环 | 知识点、题库、掌握度、LLM | 出题、批改、错因、记忆、掌握刷新 | 首页推荐、错题本、报告 | P2 → P3 → P1 → P4 → P5 |
| L2 问答链 | 知识库、会话、记忆、掌握状态 | RAG、追问、结构化回答、摘要 | 问答页、长期记忆、报告 QA 焦点 | P2 → P3 → P1 → P4 → P5 |
| L3 视频资源链 | 视频种子、知识点、外站搜索 | 爬虫、评分、去重、限流、缓存 | 出题页视频抽屉 | P2 → P3 → P1 → P4 → P5 |
| L4 论坛社交链 | 用户行为、帖子、评论、画像 | AI 回答、点赞幂等、热度、排行榜 | 论坛页、学习画像 | P2/P1 → P3 → P1 → P4 → P5 |
| L5 报告生成链 | 答题、错题、记忆、论坛、掌握度 | 聚合统计、计划、画像、导出 | 报告页、PDF/打印 | P2 → P3 → P1 → P4 → P5 |

## 5. RAG、ChromaDB 与 LangGraph 设计

### 5.1 ChromaDB Collection

P3 实现三个 collection：

```text
knowledge_base_408      # 408 知识库 Markdown chunk
user_memory_vector      # 用户长期记忆
mistake_summary         # 错题摘要和错因
```

### 5.2 Chunk 元数据

```json
{
  "doc_id": "os_page_replacement_001",
  "source": "knowledge_docs/操作系统_页面置换算法.md",
  "subject": "操作系统",
  "knowledge_point": "页面置换算法",
  "chunk_type": "concept|exam_focus|steps|common_mistakes|example",
  "keywords": ["LRU", "FIFO", "缺页次数"]
}
```

### 5.3 RAG 检索顺序

```text
1. 识别 subject + knowledge_point
2. 查 ChromaDB knowledge_base_408
3. 查 ChromaDB user_memory_vector
4. 查 ChromaDB mistake_summary
5. 查 MySQL knowledge_mastery
6. 拼 prompt
7. LLM 生成结构化回答
8. 校验 JSON
9. 失败则 fallback
```

### 5.4 LangGraph 优先改造顺序

```text

```

```text

```


### 5.5 Agent 步骤标准

每个 Agent 返回 `agent_steps`：

```json
[
  {
    "name": "识别目标",
    "input_summary": "用户询问 LRU 缺页次数",
    "output_summary": "操作系统/页面置换算法",
    "duration_ms": 12,
    "status": "success"
  }
]
```

所有步骤必须有：

- `name`
- `input_summary`
- `output_summary`
- `duration_ms`
- `status`

## 7. 核心规则公式

### 7.1 今日计划选择

输入：

```text
knowledge_mastery
最近 10 条 answer_record
最近 5 条 mistake
active user_memory
高频 knowledge_point
```

优先级：

```text
1. final_status = 薄弱点 且 weak_score 最高
2. final_status = 不会 且 wrong_count 最高
3. final_status = 不熟 且距离上次练习 > 3 天
4. is_high_frequency = true 且 final_status = 未学
5. 四科基线诊断
```

输出：

```json
{
  "title": "今天优先攻克\n页面置换算法",
  "reason": "最近 3 次 LRU 缺页统计中出现规则混淆。",
  "subject": "操作系统",
  "knowledge_point": "页面置换算法",
  "mode": "薄弱点强化",
  "count": 3,
  "difficulty": "中等"
}
```

### 7.2 推荐评分

```text
recommend_score =
  status_weight * 0.40
  + recency_weight * 0.20
  + frequency_weight * 0.15
  + weak_score_weight * 0.15
  + interval_weight * 0.10
```

权重：

```text
status_weight:
薄弱点=100，不会=80，不熟=60，掌握=20，未学=10

recency_weight:
max(0, 30 - days_since_answer) / 30 * 100

frequency_weight:
min(wrong_count, 10) / 10 * 100

weak_score_weight:
min(weak_score, 20) / 20 * 100

interval_weight:
max(0, min(days_since_recommend, 7)) / 7 * 100
```

### 7.3 掌握度与题本分离

知识点级 `knowledge_mastery.final_status`：

```text
未学
掌握
不熟
不会
薄弱点
```

`weak_score`：

```text
wrong_count * 3
+ unfamiliar_count * 2
+ unknown_count * 4
+ ocr_mistake_count * 3
+ concept_error_count * 2
+ rule_confusion_count * 2
+ qa_count * 1
+ forum_count * 1
- correct_count * 1
- mastered_count * 2
```

判断优先级：

```text
无任何学习行为 -> 未学
weak_score >= 10 或 repeated_error_type >= 2 -> 薄弱点
unknown_count >= 1 或 correct_rate < 50% 或 continuous_wrong_count >= 2 -> 不会
unfamiliar_count >= 1 或 50% <= correct_rate < 80% 或 wrong_count in [1,2] -> 不熟
total_answer_count >= 3 且 correct_rate >= 80% 且 weak_score <= 2 -> 掌握
其他 -> 不熟
```

题目级题本状态：

```text
点击“不熟” -> mistake.mastery_status = 不熟，进入不熟题本
点击“不会” -> mistake.mastery_status = 不会，进入不会题本
点击“掌握” -> mistake.status = resolved，移出题本
同题复练答对 2 次 -> 不会降为不熟
同题连续答对 3 次 -> 移出题本
```

### 7.4 错因推荐

```text
recommended_causes =
  固定 ["审题错误", "计算错误"]
  + knowledge_point.common_mistakes 提取 1-2 个
  + 用户最近 20 条 mistake 高频错因 1 个
  + LLM 从反馈提取 0-1 个
```

输出 4-5 个，前两个在前端标记“推荐”。

### 7.5 视频评分

```text
quality_score =
  platform_weight * 0.30
  + relevance_weight * 0.40
  + popularity_weight * 0.20
  + freshness_weight * 0.10
```

限流：

```text
每个 knowledge_point 每小时最多爬 1 次
每个 IP 每小时最多手动触发 5 次
已有 >= 3 条视频不爬
已有 1-2 条返回已有并后台补充
0 条返回同科目 fallback 并后台补充
```

### 7.6 论坛排行榜

```text
daily_score =
  post_count_today * 10，上限 50
  + comment_count_today * 5，上限 40
  + likes_received_today * 3
  + ai_usage_today * 8，上限 40
  + checkin_bonus 5
  + seven_day_streak_bonus 10
```

### 7.7 帖子热度

```text
hot_score =
  like_score * 0.35
  + comment_score * 0.30
  + freshness_score * 0.25
  + author_reputation * 0.10
```

### 7.8 报告画像标签

最多 3 个标签：

```text
连续学习 N 天
步骤遗漏占比 > 30% -> 分步型学习者
计算错误占比 > 25% -> 计算易遗漏
概念理解错误占比 > 30% -> 概念型学习者
审题错误占比 > 20% -> 审题要仔细
规则混淆占比 > 30% -> 规则混淆型
60% 以上答题在 8:00-12:00 -> 晨型学习者
60% 以上答题在 20:00-24:00 -> 夜间学习者
```
