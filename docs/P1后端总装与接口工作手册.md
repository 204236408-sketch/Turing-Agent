# P1 后端总装与接口工作手册

> 角色定位：后端总装、接口编排、MySQL 切换、云部署。  
> 你的上游是 P2 和 P3，下游是 P4 和 P5。你负责把数据和 Agent 能力包装成稳定、可测、前端可直接调用的 API。

## 1. 你负责的大模块

| 模块 | 负责内容 | 关键文件 |
|---|---|---|
| 路由总装 | 注册、补齐、统一所有 API | `backend/main.py`、`backend/routers/*.py` |
| 接口契约 | 统一成功/失败响应、参数校验、错误码 | `backend/utils/response.py`、`backend/schemas.py` |
| MySQL 切换 | 让后端生产环境使用 MySQL | `backend/config.py`、`backend/database.py`、`backend/requirements.txt`、`deploy/docker-compose.yml` |
| 视频接口 | 视频推荐、手动爬取、点击统计 | `backend/routers/video_router.py`、`backend/routers/question_router.py` |
| 题目收藏 | 收藏/取消收藏/收藏状态 | `backend/models.py`、`backend/routers/question_router.py` |
| 错题接口 | 错题详情、相似错题、题本筛选 | `backend/routers/mistake_router.py` |
| 论坛接口 | 搜索分页、点赞收藏幂等、排行榜 | `backend/routers/forum_router.py` |
| 报告导出 | PDF 或打印降级导出 | `backend/routers/report_router.py` |
| 云部署 | Nginx、systemd、环境变量、日志 | `deploy/*`、`docs/部署说明.md` |

## 2. 先后顺序和交接关系

```text
P2 先交付：schema、种子数据、MySQL 表结构
P3 先交付：Agent 函数返回结构、爬虫服务、Chroma 降级能力
P1 再交付：API 路由、响应格式、异常处理
P4 后续接收：接口地址、请求体、返回字段
P5 最后验收：接口测试、全链路测试、部署测试
```

你每天都要问上游两个问题：

1. P2 今天新增或修改了哪些表和字段？
2. P3 今天哪个 Agent 函数的返回字段已经稳定？

你每天都要交给下游三样东西：

1. 接口地址。
2. 请求体字段。
3. 返回 JSON 示例和错误情况。

## 3. 统一接口规则

所有接口必须遵守统一响应格式。

成功：

```json
{
  "ok": true,
  "message": "success",
  "data": {}
}
```

失败：

```json
{
  "ok": false,
  "error": {
    "code": "QUESTION_NOT_FOUND",
    "message": "题目不存在"
  }
}
```

HTTP 状态码规则：

- `200`：成功。
- `400`：业务参数错误。
- `401`：未登录或 token 失效。
- `404`：资源不存在。
- `422`：字段校验失败。
- `500`：真正未预期错误，不能用 500 表达普通业务失败。

## 4. 五天工作安排

### 第 1 天：接口盘点和 MySQL 生产配置

#### 要做什么

1. 打开 `backend/main.py`，记录当前注册了哪些 router。
2. 打开 `backend/routers/*.py`，把接口分成三类：
   - 已完成可复用。
   - 半成品需要深化。
   - 仍是 Mock 或只返回 toast 文案。
3. 检查 `backend/utils/response.py`，确认成功/失败响应格式统一。
4. 修改 `backend/requirements.txt`，补：
   - `pymysql`
   - `cryptography`
   - `apscheduler`
5. 修改 `deploy/docker-compose.yml`，增加 MySQL 服务，不再只写 SQLite。
6. 配合 P2 对齐 `database/schema.sql`，确认后端模型和 MySQL 表一致。

#### 交付给谁

- 交给 P4：接口清单和第一版接口契约。
- 交给 P5：哪些接口当天可以开始测。
- 交给 P2：后端实际需要的表和字段清单。

#### 完成标准

- `docs/接口文档.md` 里至少列出：登录、首页、出题、答题、错题、视频、论坛、报告接口。
- `docker-compose.yml` 中有 MySQL 服务配置。
- 后端仍可用 SQLite 本地启动，同时配置 `MYSQL_URL` 后可切 MySQL。

### 第 2 天：视频接口和题目收藏

#### 要做什么

1. 修改 `backend/routers/video_router.py`。
   - `GET /api/videos/recommend`
   - `GET /api/videos/list`
   - 支持 `subject`、`knowledge_point`、`limit`。
2. 修改 `backend/routers/question_router.py` 的 `/{question_id}/videos`。
   - 先根据题目查 `subject` 和 `knowledge_point`。
   - 再调用视频推荐查询。
   - 不能只按 subject 查。
3. 增加视频返回字段：
   - `id`
   - `title`
   - `platform`
   - `url`
   - `reason`
   - `quality_score`
   - `crawl_triggered`
   - `fallback_used`
4. 新增题目收藏逻辑。
   - 表：`favorite_question`
   - 接口：`POST /api/questions/{question_id}/collect`
   - 再点一次可取消收藏。
   - 返回：`collected`、`favorite_count`。

#### 细化规则

视频推荐逻辑：

```text
如果该知识点已有 >= 3 条视频：
  直接按 quality_score 降序返回 top 3。

如果有 1-2 条：
  先返回已有视频，后台触发 P3 爬虫补充。

如果 0 条：
  返回同科目高分视频作为 fallback，同时后台触发爬虫。
```

收藏逻辑：

```text
同一 user_id + question_id 只能有一条收藏记录。
如果未收藏，创建记录。
如果已收藏，再次调用则删除或标记 inactive。
接口必须幂等，不允许重复插入。
```

#### 交付给谁

- 交给 P4：视频抽屉接口、收藏按钮接口。
- 交给 P5：视频和收藏测试入口。
- 依赖 P2：`video_resource` 字段和种子数据。
- 依赖 P3：爬虫服务函数可被调用，失败不抛出到用户。

#### 完成标准

- 任意题目点击推荐视频，接口返回结构稳定。
- 数据库无视频时不报错。
- 收藏题目不会重复增加。

### 第 3 天：错题详情和相似错题

#### 要做什么

1. 修改 `backend/routers/mistake_router.py` 的 `GET /api/mistakes/{mistake_id}`。
2. 返回完整错题详情：
   - 错题 id
   - 题目 id
   - 科目
   - 知识点
   - 原题
   - 选项
   - 标准答案
   - 用户答案
   - 是否正确
   - 错因类型
   - 错因说明
   - 复习建议
   - 相似错题列表
3. 新增 `POST /api/mistakes/similar`。
4. 接入 P3 的 `chroma_service`。
5. Chroma 不可用时，按 `subject + knowledge_point` 降级查询。

#### 细化规则

相似错题返回逻辑：

```text
优先级 1：ChromaDB 语义相似。
优先级 2：同 user_id + 同 knowledge_point。
优先级 3：同 subject + error_type 相同。
优先级 4：返回空数组，不报错。
```

错题本筛选：

```text
GET /api/mistakes/notebook?status=不熟
GET /api/mistakes/notebook?status=不会

排序：
weak_score 降序 → wrong_count 降序 → last_answer_time 升序。
```

#### 交付给谁

- 交给 P4：错题详情页面接口。
- 交给 P5：错题详情和相似错题测试。
- 依赖 P3：`chroma_service.search_similar_mistakes`。
- 依赖 P2：`knowledge_mastery`、`mistake`、`answer_record` 字段完整。

#### 完成标准

- 错题详情不再只返回 error_reason 和 suggestion。
- 没有 ChromaDB 也能返回同知识点错题或空数组。
- 前端能按“不熟/不会”区分题本。

### 第 4 天：论坛、排行榜和报告导出

#### 要做什么

1. 修改 `backend/routers/forum_router.py`。
2. 帖子列表支持：
   - `q` 或 `search`
   - `category`
   - `page`
   - `size`
3. 新增或完善：
   - `forum_like`
   - `forum_collect`
   - `user_daily_activity`
4. 点赞和收藏必须按用户幂等。
5. 新增排行榜接口：
   - `GET /api/forum/leaderboard?scope=daily&limit=10`
   - `GET /api/forum/leaderboard?scope=weekly&limit=10`
6. 修改 `backend/routers/report_router.py`，新增导出接口。

#### 论坛热度规则

```text
hot_score =
  like_score * 0.35 +
  comment_score * 0.30 +
  freshness_score * 0.25 +
  author_reputation * 0.10
```

#### 排行榜规则

```text
daily_score =
  今日发帖数 * 10，上限 50 +
  今日评论数 * 5，上限 40 +
  今日收到点赞数 * 3 +
  今日使用 AI 小助手次数 * 8，上限 40 +
  今日打卡 5 分，连续 7 天额外 10 分
```

#### 报告导出规则

优先 PDF：

```text
GET /api/reports/{report_id}/export?format=pdf
```

如果 PDF 库不可用：

```json
{
  "downloadable": false,
  "fallback": "print",
  "html": "<html>...</html>"
}
```

#### 交付给谁

- 交给 P4：论坛分页、排行榜、点赞收藏、报告导出接口。
- 交给 P5：论坛幂等、排行榜、报告导出测试。
- 依赖 P2：新增表。
- 依赖 P3：论坛 AI 结构化回答和 activity 写入规则。

#### 完成标准

- 同一用户重复点赞不会无限 +1。
- 排行榜能返回 daily 和 weekly。
- 报告导出失败时能走打印降级。

### 第 5 天：云部署和最终联调

#### 要做什么

1. 配置云服务器环境变量：
   - `MYSQL_URL`
   - `SECRET_KEY`
   - `SILICONFLOW_API_KEY`
   - `CORS_ORIGINS`
   - `UPLOAD_DIR`
   - `REPORT_DIR`
2. 修改或确认：
   - `deploy/nginx.conf`
   - `deploy/yantu408.service`
   - `deploy/docker-compose.yml`
3. 启动 MySQL。
4. 启动后端。
5. 配置 Nginx，把 `/api` 代理到后端。
6. 配合 P2 导入数据。
7. 配合 P5 做全链路验收。

#### 交付给谁

- 交给所有人：云端访问地址。
- 交给 P5：部署验收环境。
- 交给用户/答辩：可演示地址和部署说明。

#### 完成标准

- 云端可访问前端。
- `/api/health` 正常。
- 登录、首页、出题、答题、错题、论坛、报告都能跑。
- 后端日志可查看。
- MySQL 重启后数据不丢。

- 你有没有被 P2/P3 阻塞，是否已在当天同步会上说明。
