# P2 数据库与数据资源工作手册

> 角色定位：数据库设计、MySQL schema、种子数据、知识库、题库、视频资源、数据迁移。  
> 你是所有流水线的最上游。P3、P1、P4、P5 都依赖你的数据质量。

## 1. 你负责的大模块

| 模块 | 负责内容 | 关键文件 |
|---|---|---|
| MySQL schema | 补齐生产表结构、索引、字段 | `database/schema.sql`、`backend/models.py` |
| 知识点数据 | 408 四科知识点、父级关系、关键词、常见错因 | `backend/data/seed_knowledge_points.json` |
| 题库数据 | 选择题、综合题、判断题、解析、提示 | `backend/data/seed_questions.json` |
| 视频种子数据 | 每个高频知识点至少 2 条视频 | `backend/data/seed_video_resources.json` |
| 知识库文档 | RAG/Chroma/MySQL 降级检索所需 Markdown | `backend/data/knowledge_docs/` |
| 数据导入 | 种子数据一键导入 | `backend/scripts/seed_data.py` |
| 数据迁移 | SQLite 迁移到 MySQL | `backend/scripts/migrate_sqlite_to_mysql.py` |
| 备份 | 云端 MySQL 备份 | `deploy/backup_mysql.sh` |

## 2. 先后顺序和交接关系

```text
P2 先交付数据和表结构
P3 根据你的数据写 Agent、RAG、爬虫、评分逻辑
P1 根据你的表结构写路由和接口
P4 根据接口展示你的数据
P5 根据你的数据写测试样例
```

你每天必须交付给下游：

- 新增或修改的表。
- 新增或修改的字段。
- 种子数据格式。
- 至少一组可用于测试的真实样例。

## 3. 数据命名规则

### 3.1 科目名称固定

只能使用这四个：

- `数据结构`
- `计算机组成原理`
- `操作系统`
- `计算机网络`

不要写：

- `计组`
- `组成原理`
- `网络`
- `OS`

### 3.2 知识点名称固定

同一个知识点全项目只能有一个名称。例如：

- 使用 `页面置换算法`
- 不要混用 `页置换`、`页面置换`、`LRU算法`

### 3.3 高频字段

`is_high_frequency` 用布尔值：

```json
"is_high_frequency": true
```

不要写：

```json
"high_frequency": true
```

除非同步修改 `seed_data.py` 兼容。

## 4. 五天工作安排

### 第 1 天：补齐 schema 和知识点

#### 要做什么

1. 对照 `backend/models.py` 重写 `database/schema.sql`。
2. 补齐当前 schema 缺少的表：
   - `user_profile`
   - `subject`
   - `forum_category`
   - `forum_comment`
   - `forum_checkin`
   - `conversation`
   - `conversation_message`
   - `favorite_question`
   - `forum_like`
   - `forum_collect`
   - `user_daily_activity`
   - `user_pending_recommendation`
   - `knowledge_document`
   - `video_crawl_log`
3. 扩充 `backend/data/seed_knowledge_points.json`。
4. 至少准备 40 个知识点。

#### 知识点 JSON 格式

```json
{
  "subject": "操作系统",
  "name": "页面置换算法",
  "parent_name": "内存管理",
  "level": 3,
  "content": "掌握 FIFO、LRU、OPT、Clock 等页面置换算法的缺页次数计算与适用场景。",
  "common_mistakes": "命中后未更新 LRU 顺序；初始装入缺页漏算；混淆页面号和物理块号。",
  "keywords": "LRU,FIFO,OPT,Clock,缺页中断,页框,虚拟内存",
  "is_high_frequency": true
}
```

#### 交付给谁

- 交给 P3：知识点、常见错因、关键词。
- 交给 P1：schema 和新增字段。
- 交给 P5：可测试的知识点样例。

#### 完成标准

- MySQL schema 能建库。
- 40 个知识点字段完整。
- 每科至少 8 个知识点。
- 高频知识点至少 20 个。

### 第 2 天：补题库和视频种子

#### 要做什么

1. 扩充 `backend/data/seed_questions.json`。
2. 每个高频知识点至少 3 道题。
3. 每题必须有：
   - 科目
   - 知识点
   - 难度
   - 题型
   - 题干
   - 选项或子问题
   - 标准答案
   - 解析
   - 三层提示
4. 扩充 `backend/data/seed_video_resources.json`。
5. 每个高频知识点至少 2 条视频。

#### 题目 JSON 格式

```json
{
  "subject": "操作系统",
  "knowledge_point": "页面置换算法",
  "difficulty": "中等",
  "question_type": "选择题",
  "variant_type": "choice",
  "question_text": "某进程访问页面序列为 1,2,3,4,1,2,5，分配 3 个页框，采用 LRU 算法，缺页次数为多少？",
  "options": ["A. 5", "B. 6", "C. 7", "D. 8"],
  "standard_answer": "C",
  "explanation": "初始装入 1、2、3 均缺页，访问 4 替换 1，随后 1、2、5 均产生缺页，共 7 次。",
  "hints": ["先列出页框变化表。", "命中后也要更新最近使用顺序。", "初始装入也算缺页。"],
  "source": "seed"
}
```

#### 视频 JSON 格式

```json
{
  "subject": "操作系统",
  "knowledge_point": "页面置换算法",
  "title": "408 操作系统 页面置换算法 FIFO LRU OPT 详解",
  "platform": "Bilibili",
  "url": "https://www.bilibili.com/video/BVxxxx",
  "reason": "从缺页中断原理讲起，适合补基础并对比算法差异。",
  "quality_score": 88,
  "duration": "18:32",
  "author": "408考研UP主",
  "crawl_source": "seed"
}
```

#### 交付给谁

- 交给 P3：题库用于 Agent fallback，视频用于评分和爬虫补充。
- 交给 P1：视频表字段和导入规则。
- 交给 P4：视频展示字段样例。
- 交给 P5：题目和视频测试样例。

#### 完成标准

- 至少 60 道题。
- 至少 40 条视频。
- 视频链接能打开。
- 每条视频都有推荐理由。

### 第 3 天：知识库文档和行为表

#### 要做什么

1. 新建目录 `backend/data/knowledge_docs/`。
2. 每个高频知识点写一份 Markdown。
3. 每份 Markdown 包含：
   - 核心概念
   - 408 常考方式
   - 常见错因
   - 标准回答模板
4. 在 schema 和模型中补：
   - `user_daily_activity`
   - `user_pending_recommendation`
   - `knowledge_document`

#### 知识库 Markdown 格式

```md
# 页面置换算法

科目：操作系统
父级：内存管理
关键词：LRU、FIFO、OPT、Clock、缺页中断

## 核心概念
页面置换算法用于在页框不足时选择被换出的页面。

## 408 常考方式
1. 给定访问序列计算缺页次数。
2. 比较 FIFO、LRU、OPT 的替换结果。

## 常见错因
1. 初始装入缺页漏算。
2. LRU 命中后未更新最近使用顺序。

## 标准回答模板
先列页框变化，再统计缺页次数，最后说明替换依据。
```

#### 交付给谁

- 交给 P3：RAG 导入源文件。
- 交给 P1：`knowledge_document` 表结构。
- 交给 P5：知识库检索测试样例。

#### 完成标准

- 至少 20 份 Markdown 知识文档。
- 每份文档文件名清晰，例如 `操作系统_页面置换算法.md`。
- 文档中的知识点名称必须和 JSON 完全一致。

### 第 4 天：迁移脚本和一键导入

#### 要做什么

1. 新建 `backend/scripts/migrate_sqlite_to_mysql.py`。
2. 读取 `turing408.db`。
3. 连接 MySQL。
4. 按顺序导入：
   - 用户和资料表
   - 知识点和题目表
   - 答题、错题、记忆表
   - 论坛和报告表
   - 视频表
5. 修改 `backend/scripts/seed_data.py`，支持一键导入所有 JSON。
6. 更新 `database/demo_data.sql`，补演示账号、论坛分类和基础帖子。

#### 导入顺序规则

```text
先导入无依赖表：
user, user_profile, subject, knowledge_point, forum_category

再导入业务基础表：
question_generation_session, question, video_resource

再导入用户行为表：
answer_record, knowledge_mastery, mistake, user_memory

最后导入展示和社区表：
conversation, conversation_message, forum_post, forum_comment, report
```

#### 交付给谁

- 交给 P1：可用于云端 MySQL 的导入脚本。
- 交给 P5：迁移前后数量校验方式。

#### 完成标准

- MySQL 空库可以一键导入演示数据。
- 每张表导入后输出数量。
- SQLite 到 MySQL 数量一致或说明差异。

### 第 5 天：云端数据验收和备份

#### 要做什么

1. 配合 P1 在云服务器执行建库和导入。
2. 新建 `deploy/backup_mysql.sh`。
3. 备份脚本每天导出 MySQL。
4. 保留最近 7 天备份。
5. 输出数据验收表。

#### 数据验收表至少包含

| 项目 | 最低数量 |
---|---:|
| 用户 | 1 |
| 科目 | 4 |
| 知识点 | 40 |
| 题目 | 60 |
| 视频 | 40 |
| 知识库文档 | 20 |
| 论坛分类 | 4 |
| 演示帖子 | 5 |

#### 完成标准

- 云端 MySQL 中基础数据不为空。
- 后端重启后数据仍在。
- 备份脚本可以手动执行成功。

## 5. 关键规则和逻辑

### 掌握度相关数据

你不直接写 final_status 逻辑，但要保证字段支持：

- `total_answer_count`
- `correct_count`
- `wrong_count`
- `unfamiliar_count`
- `unknown_count`
- `mastered_count`
- `ocr_mistake_count`
- `qa_count`
- `forum_count`
- `weak_score`
- `last_answer_time`

### 推荐数据

推荐依赖：

- 知识点高频标记。
- 错题次数。
- 上次答题时间。
- 用户记忆。
- 题目和视频是否足够。

所以种子数据不能只写名字，必须写内容、错因和关键词。

### 视频资源质量

人工种子视频必须优先保证有效性，不追求数量灌水。每条都要能解释“为什么推荐”。

## 6. 每日自查清单

- 科目和知识点名称是否统一？
- 新增表是否同步到 `database/schema.sql`？
- 是否给 P3/P1/P5 说明了字段变化？
